# 訓練醫院容額填報系統設計文檔

## 概述

本文檔說明「容額填報」功能中的「訓練醫院容額」（Quota）模組的核心設計邏輯、資料結構和流程。

---

## 核心概念

### 申請主體（Institution Entity）

「申請主體」是一個邏輯抽象層，代表實際進行申請的單位。申請主體可以是：

1. **單一機構**：一間醫療機構本身（如「台大醫院」）
2. **機構組合**：多間醫療機構合併而成（如「北部聯合訓練中心」由「台大醫院」+「台北榮總」組成）

### 資料歸屬

| 資料欄位 | 歸屬對象 |
|---------|---------|
| 效期起始／結束年度 | 申請主體（單一或組合） |
| 容額（前年度核定、可收訓容額、建議分配） | 整個申請案 |
| 醫事機構代碼、醫院名稱 | 單一醫療機構或申請主體名稱 |

**關鍵原則**：效期跟著「醫院名稱」（即申請主體），而非跟著個別的醫療機構代碼。

---

## 申請方式

### 單一機構申請
- **結構**：1 個申請主體（可為單一機構或機構組合）
- **例子**：
  - 台大醫院單獨申請
  - 「北部聯合訓練中心」（由多個機構組成）單獨申請

### 聯合申請
- **結構**：1 個主訓主體 + N 個合作主體（每個主體都可為單一機構或機構組合）
- **例子**：
  ```
  聯合申請案例：
  ├─ 主訓主體：「林口長庚醫院」（單一機構）
  ├─ 合作主體 1：「台北榮總」（單一機構）
  └─ 合作主體 2：「北部聯合訓練中心」（機構組合）
       └─ 組成：台大醫院 + 三軍總醫院
  ```

**容額歸屬**：聯合申請中的容額數字屬於整個申請案，不分主訓/合作機構。

## 資料結構

### InstitutionEntity（申請主體）

```typescript
interface InstitutionEntity {
  id: string                        // 主體唯一識別碼
  type: "single" | "merged"         // 主體類型
  
  // 單一機構用
  hospitalCode?: string             // 醫療機構代碼
  
  // 機構組合用
  mergedName?: string               // 組合名稱（如「北部聯合訓練中心」）
  mergedHospitalCodes?: string[]    // 組成機構的代碼列表
  
  // 效期（兩種主體都有）。民國年份字串，UI 以下拉選單選取（108～127）
  expiryStartYear: string           // 效期起始年度（如「113」）
  expiryEndYear: string             // 效期結束年度（如「115」）
}
```

### QuotaFormValues（申請表單資料）

```typescript
interface QuotaFormValues {
  applicationMode: "single" | "joint"  // 申請方式
  
  // 主訓機構（單一/聯合都有）
  mainEntity: InstitutionEntity | null
  
  // 合作機構（聯合申請時）
  partnerEntities: InstitutionEntity[]
  
  // 容額設定
  prevQuota: string                   // 前年度核定容額
  isNewApplication: boolean           // 是否為新申請（無前年度核定）
  quotaLimit: string                  // 容額數字（整個申請案共用）
  currentQuota: string                // 可收訓容額
  
  // 備註
  note: string
}
```

## UI/UX 流程

### 新增訓練醫院容額頁面

#### Step 1：選擇申請方式
```
[選擇申請方式]
├─ 單一機構申請
└─ 聯合申請
```

#### Step 2：建立主訓機構/合作機構

**Dialog 分層流程**：

```
第一層：選擇機構類型
  ├─ 單一機構：選擇一間醫療機構
  └─ 機構組合：將多間機構組合成一個主體
        ↓ 選擇其一
        
第二層：選擇醫院（可內部滾動）
  ├─ 機構組合名稱輸入（僅組合時）
  ├─ 搜尋列 & 縣市篩選
  ├─ 醫院清單（高度限制 240px，可滾動）
  └─ 已選擇機構標籤
        ↓ 選完醫院
        
第三層：設定效期（藍色背景突出）
  ├─ 效期起始年度：[下拉選單]
  ├─ 效期結束年度：[下拉選單]
  └─ 按鈕：[上一步] [取消] [確認新增]
```

#### Step 3：回到表單（機構卡片 - 內聯編輯）

已選擇的申請主體顯示為**可編輯卡片**：

```
┌─────────────────────────────────┐
│ 主訓機構                         │
├─────────────────────────────────┤
│ 機構名稱：台大醫院              │
│                                 │
│ 效期起始年度：[113 年] ▼         │
│ 效期結束年度：[115 年] ▼         │
│                                 │
│ [編輯機構]  [×]                 │
└─────────────────────────────────┘
```

**卡片功能**：
- **效期起始／結束年度**：直接在卡片上編輯（內聯編輯）
- **編輯機構**：僅機構組合時顯示，點擊進入 Dialog 第一層重新配置組成
- **×**：刪除此申請主體

## 極端但合法的場景

系統設計支援以下複雜場景：

### 場景 1：單一申請中的組合
```
單一機構申請
└─ 申請主體：「南部聯合訓練中心」
   ├─ 台大醫院
   ├─ 台北榮總
   └─ 三軍總醫院
```

### 場景 2：聯合申請中多個組合
```
聯合申請
├─ 主訓主體：「北部聯合訓練中心」
│  ├─ 台大醫院
│  └─ 台北榮總
├─ 合作主體 1：中國醫藥大學附醫（單一）
└─ 合作主體 2：「中部聯合訓練中心」
   ├─ 中國醫藥大學附醫
   └─ 台中榮總
```

### 場景 3：同一機構的多重身份
同一醫療機構（如「台大醫院」）可同時：
- 作為某個單一申請的主體
- 作為聯合申請的主訓主體
- 作為另一個單一申請中組合主體的組成機構
- 作為第三個聯合申請的合作機構組合的組成機構

**無衝突**，系統允許此情況。

## 列表呈現（容額列表）

### 申請案展示方式

| 類型 | 標記 | 列數 | 醫院名稱欄位顯示 |
|------|-----|------|----------------|
| 單一機構申請 | 無 | 1 列 | 機構名稱或組合名稱 |
| 聯合申請（主訓列） | [聯] | 多列 | 主訓機構名稱 |
| 聯合申請（合作列） | 無 | 多列 | 合作機構名稱 |

### 機構組合在列表上
- **顯示**：組合的名稱（如「北部聯合訓練中心」）
- **醫事機構代碼**：組合名稱時自動生成或使用特殊標識（如 `merged-001`）
- **效期**：跟著組合主體，不展開顯示組成機構的各自效期
- **標記**：無特殊標記（因為「機構組合」不是申請方式，而是申請主體的一種形式）

## 用詞規範

### 重點：「申請主體」不出現在介面上

| 位置 | 用詞 | 範例 |
|------|-----|------|
| 列表欄位 | 「醫院名稱」 | 「台大醫院」或「北部聯合訓練中心」 |
| 表單標籤 | 「主訓機構」、「合作機構」 | - |
| Dialog 類型選擇 | 「單一機構」、「機構組合」 | - |
| Dialog 標題 | 「建立機構組合」、「編輯機構組合」 | - |
| 卡片按鈕 | 「編輯機構」、「編輯機構組合」 | - |

**禁止用語**：「主體」、「申請主體」、「合併主體」、「合併申請」（申請方式只有「單一」和「聯合」）

## 技術實作重點

### 元件架構

```
app/filing/quota/
├─ new/page.tsx              # 新增頁面（取 params、呼叫表單）
├─ [id]/page.tsx             # 編輯頁面（載入 mock、呼叫表單）
└─ 

components/filing/
├─ quota-form.tsx            # 共用表單（申請邏輯、驗證、UI）
├─ institution-entity-selector.tsx  # 機構選擇器（分層 Dialog）
└─ hospital-multi-select.tsx # 醫院多選元件
```

### 資料流向

```
頁面（new/[id]）
  ↓ 傳入 mode、initialData
  ↓
共用表單元件（quota-form.tsx）
  ├─ 管理 applicationMode、entities 等 state
  ├─ 呈現申請方式選擇
  ├─ 呈現主訓/合作機構卡片
  │   └─ 支援內聯編輯效期
  │   └─ 支援「編輯機構」按鈕進 Dialog
  │
  └─ 機構選擇器（institution-entity-selector.tsx）
      ├─ Dialog 第一層：選擇類型
      ├─ Dialog 第二層：選擇醫院
      └─ Dialog 第三層：設定效期
```

### 驗證規則

**建立機構組合時**：
- 機構組合名稱不為空
- 至少選擇 2 間醫療機構
- 效期起始與結束年度皆須選取

**填寫容額申請表單時**：
- 主訓機構必選
- 聯合申請時，至少新增 1 個合作機構
- 容額數字：1-50 範圍
- 可收訓容額：1-50 範圍

### 編輯與重複使用

- **每次申請都重新建立**：機構組合不複用，每次申請時都需要重新建立（簡化後端實作）
- **允許修改組成**：已選機構可點擊「編輯機構」重新配置
- **同機構多身份**：同一醫療機構允許在不同申請案中扮演不同角色

## 與後端整合注意事項

### API 契約

**新增/編輯容額申請**：
```
POST /api/filing/quota
PATCH /api/filing/quota/:id

Body: QuotaFormValues {
  applicationMode,
  mainEntity: InstitutionEntity,
  partnerEntities: InstitutionEntity[],
  prevQuota,
  isNewApplication,
  quotaLimit,
  currentQuota,
  note
}
```

### 資料庫考慮

1. **主體表**：記錄每個申請主體的組成
   - `id`, `type`, `hospitalCode` (單一時), `mergedName`, `mergedHospitalCodes` (組合時)

2. **申請案表**：記錄申請案的高層資訊
   - `id`, `applicationMode`, `mainEntityId`, `partnerEntityIds`, `quotaLimit`, `currentQuota`, `createdAt`, `updatedAt`

3. **效期繼承**：效期儲存在主體表，申請案透過主體 ID 查詢
